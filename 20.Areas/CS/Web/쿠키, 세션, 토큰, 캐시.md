- **쿠키**: 사용자에 의해 조작되어도 크게 문제 되지 않을 정보를 브라우저에 저장
- **세션**: 인증에 대한 정보를 서버에 저장
- **토큰**: 인증에 대한 정보를 사용자가 저장
- **캐시**: 한 번 전송받은 데이터를 저장해 놨다가 필요할 때 꺼내 쓰기 기능

## 🍪 쿠키, 세션, 토큰, 캐시 보완 정리

### 1. 쿠키 (Cookie)
- **보완점:** 쿠키는 단순히 '정보 저장'을 넘어, 서버와 클라이언트 사이의 **통신 상태를 유지**하는 매개체입니다.
- **핵심 특징:** * **만료 기한:** 브라우저를 닫아도 유지되게 설정할 수 있습니다. (Persistent Cookie)
    - **자동 전송:** 브라우저가 서버에 요청을 보낼 때마다 HTTP 헤더에 담겨 **자동으로** 전송됩니다.
    - **보안 취약점:** 사용자 기기에 직접 저장되므로 민감 정보(비밀번호 등)를 담아서는 안 됩니다.

### 2. 세션 (Session)
- **보완점:** 세션은 독립적으로 존재하지 않고 보통 **쿠키를 빌려** 씁니다.
- **핵심 특징:**
    - **동작 방식:** 서버는 '세션 ID'를 생성해 쿠키에 담아 보냅니다. 브라우저는 이 ID만 들고 있다가 요청 시 제출합니다.
    - **장점:** 실제 정보는 서버 메모리나 DB에 있으므로 쿠키보다 안전합니다.
    - **단점:** 사용자가 많아지면 서버 메모리 부하가 커집니다. (서버의 확장성이 떨어짐)

### 3. 토큰 (Token / JWT)
- **보완점:** 세션의 '서버 부하' 문제를 해결하기 위해 등장한 **무상태(Stateless)** 방식입니다.
- **핵심 특징:**
    - **자체 포함(Self-contained):** 토큰 그 자체가 "나는 누구이며, 언제까지 유효하다"라는 정보를 암호화해서 들고 있습니다.
    - **서버의 편안함:** 서버는 토큰이 유효한지만 검증하면 될 뿐, 별도의 세션 DB를 뒤질 필요가 없습니다. (현대 웹 서비스, 모바일 앱에서 주로 사용)

### 4. 캐시 (Cache)
- **보완점:** 쿠키/세션/토큰이 **'누구인가(인증)'** 에 집중한다면, 캐시는 **'얼마나 빠른가(성능)'** 에 집중합니다.
- **핵심 특징:**
    - **종류:** 브라우저 캐시(로컬 이미지, CSS 등)뿐만 아니라 서버 캐시(Redis 등), CDN 캐시 등 다양합니다.
    - **주의사항:** 데이터가 업데이트되었는데 캐시가 남아있으면 사용자는 '옛날 화면'을 보게 됩니다. 이를 위해 캐시 무효화(Invalidation) 전략이 중요합니다.%%  %%
