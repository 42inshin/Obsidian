**: 리소스 로딩 속도 최적화를 위한 저장소**

### 1. 클라이언트 캐시 (Browser Cache)

**우리 브라우저가 직접 관리하는 캐시**

- **로컬 캐시:** 한 번 다운로드한 로고 이미지, CSS 스타일 시트, JS 파일 등을 하드디스크에 저장해둡니다. 다음에 같은 사이트에 접속할 때 서버에 요청하지 않고 내 컴퓨터에서 바로 꺼내 써서 **속도가 엄청나게 빠릅니다.**

- **메모리 캐시:** RAM에 임시 저장되는 데이터로, 브라우저를 닫으면 보통 사라지지만 속도는 가장 빠릅니다.


### 2. 서버 사이드 캐시 (Server Cache)

**서버가 일을 덜 하기 위해 사용하는 캐시**

- **데이터베이스 캐싱:** 똑같은 SQL 쿼리가 반복되면 DB까지 가지 않고 메모리(예: Redis)에서 바로 결과를 반환합니다.

- **CDN (Content Delivery Network):** 전 세계 곳곳에 서버를 두고, 사용자와 가장 가까운 서버에서 이미지나 영상을 전달합니다. 이것도 일종의 거대한 캐시 시스템입니다.

---

## 캐시 제어

### 1. 브라우저 캐시 제어 (HTTP Header)

브라우저에게 "이 데이터는 얼마나 오래 보관해도 돼?"라고 명령을 내리는 방식입니다. 주로 HTTP 응답 헤더를 통해 조절합니다.

#### 주요 헤더와 전략

- **`Cache-Control`**: 가장 중요한 헤더입니다.
    
    - `max-age=3600`: 1시간 동안 이 데이터를 캐시해도 됨.
        
    - `no-cache`: 캐시는 하되, 사용할 때마다 서버에 "이거 써도 돼?"라고 물어봐(검증해).
        
    - `no-store`: 절대 저장하지 마(민감한 정보).
        
- **`ETag` (Entity Tag)**: 데이터의 "지문" 같은 값입니다.
    
    - 데이터가 조금이라도 바뀌면 ETag 값이 변합니다. 브라우저는 서버에 요청할 때 이 값을 보내서 "내 지문이랑 같아?"라고 묻고, 같다면 서버는 데이터를 다시 보내지 않고 **304 Not Modified** 응답만 보냅니다. (네트워크 대역폭 절약)


### 2. 서버 사이드 캐시 (Redis)

서버 내부에서 DB 조회를 줄이기 위해 사용하는 메모리 기반의 캐시 시스템입니다.

#### 왜 Redis인가?

- **속도**: 하드디스크가 아닌 메모리(RAM)에 데이터를 저장하므로 조회 속도가 압도적으로 빠릅니다.
    
- **Key-Value 구조**: `User:123` : `{name: "Gemini"}` 식의 간단한 구조로 데이터를 빠르게 읽고 씁니다.
    
- **데이터 타입**: 리스트, 셋, 해시 등 다양한 형식을 지원하여 복잡한 데이터도 캐싱하기 좋습니다.
    

#### 주요 캐싱 전략

- **Look Aside (가장 일반적)**:
    
    1. 서버가 데이터를 찾을 때 먼저 Redis(캐시)를 확인합니다.
        
    2. 없으면 DB에서 데이터를 가져온 뒤, 다시 Redis에 저장해둡니다.
        
- **Write Back**:
    
    1. 데이터를 일단 Redis에 다 저장해두고, 나중에 한꺼번에 DB에 반영합니다. (로그 데이터처럼 쓰기 작업이 엄청나게 많을 때 사용)
